# Magpie v0.4.1 计划：Stability & Performance

> 状态：草案（v0.4.0 发布前定稿，发布后启动）
> 主题：稳定与性能内功，不引入新功能
> 周期：v0.4.0 发布后 2~3 周
> 路线图归属：方案 A（迁移巩固）+ 方案 C（性能/稳定性专项），见 v0.4.0 发布前的讨论。

---

## 0. 不做的（明确划线）

- ❌ Snippets / OCR / Smart Actions（留 0.5.0）
- ❌ 主题 / UI 美化（除非有 issue 明确要求）
- ❌ 新协议 / 新数据结构（保 v0.4.0 的兼容承诺）
- ❌ macOS 适配（本机不可 build，无意义）
- ❌ 触碰内部兼容字段：`<!--TIEZ_RICH_IMAGE:`、`tiez-sync`、`tiez.log`、`tiez_xxx` localStorage、`tiez/tiez_xxx` MQTT topic

## 1. A 类：迁移巩固 / Hotfix（基于实际反馈）

| 编号 | 内容 | 优先级 |
|------|------|--------|
| A1 | v0.4.0 上线后**观察期收集 issue**，针对实际问题修 | P0（看反馈） |
| A2 | **卸载体验改善**：检测到 Magpie 进程时弹"是否关闭？"提示框，而非 `taskkill /F` 硬杀 | P1 |
| A3 | **迁移日志可见性**：在 `tiez.log` 头部输出 `[MIGRATION v040] from=... to=... db_size=...`，便于用户/排查 | P1 |
| A4 | **更新检查错误分类**：失败时把网络层（DNS / TLS）错误分类，给出"GitHub 暂时不可达"等中文提示，不再直接抛英文 | P2 |
| A5 | **datapath.txt 健康检查**：首次启动若 datapath 指向的盘符不存在则回退默认目录并写日志 | P2 |

## 2. C 类：性能 / 稳定性内功

| 编号 | 内容 | 优先级 |
|------|------|--------|
| C1 | **基准测试套件**：写一个 `benches/` 目录，criterion 测三个核心 op：`get_history(0, 100)`、`search('keyword', 1000条)`、`insert_clipboard()`。出 `perf-baseline.md` | P0 |
| C2 | **大列表实测**：mock 5000 / 20000 / 100000 条数据看 `VirtualClipboardList` 滚动 fps、内存占用、搜索响应 | P0 |
| C3 | **内存泄漏排查**：跑 24 小时持续监听剪贴板（脚本每秒 setText），看 RSS 是否单调上涨。重点查 `richTextSnapshot` 的缓存 Map 和 sensitive_align 队列 | P1 |
| C4 | **启动速度优化**：实测 cold start ms。可能优化点：① `init_db` 的 schema 检查走异步、② 主题应用前先显示窗口骨架、③ `start_services` 改并行 `tokio::join!` | P1 |
| C5 | **Win+V 注册健壮性 / 接管系统快捷键**：当用户在设置中选 `Win+V` 作主快捷键时，Magpie 当前会因系统级 Win+V 占用直接报「快捷键不可用」（v0.4.0 已知问题）。修复方案：① 在 ClipboardSettingsGroup 恢复"让 Magpie 接管 Win+V"开关；② 注册失败时检测是否因系统占用，弹一次性提示「需要释放系统 Win+V，是否启用接管？」；③ 接管时调 `trigger_registry_win_v_optimization(true)` 写 `DisabledHotkeys` 并提示重启资源管理器；④ 关闭时还原；⑤ 同时检测 PowerToys / Ditto 占用 Win+V 的场景给出说明 | P0（v0.4.0 实际反馈）|
| C6 | **Panic 兜底**：① 全局 `panic_hook` 写到 `tiez.log`、② grep 一遍 `.unwrap()` 把可恢复的换 `.unwrap_or_else(...)`、③ 主线程 panic 时尝试优雅落盘 db | P1 |
| C7 | **Cargo opt 升级**：试 `opt-level = "z"` + `strip = "symbols"`，看体积能否再降。**前提**：跑回归测试确认没引入 bug | P2 |
| C8 | **CI 加 Rust cache**：`Swatinem/rust-cache@v2`，预期 CI 时间从 30+ 分钟 → 10~15 分钟 | P2（cosmetic） |

## 3. 推荐工作节奏

```
Week 0:   v0.4.0 发布
Week 1:   开发 C1 + C2（基准测试 + 大列表实测）→ 出 perf-baseline.md
Week 1-2: 收集用户/自用 issue，挑出 A 类 hotfix 列表
Week 2:   C3 + C4（内存 + 启动速度）→ 优化措施
Week 2-3: C6 + 卸载改善（A2）→ 把 panic 兜底和卸载提示弄稳
Week 3:   C7 + C8（可选，时间剩余的话）+ 双盲审计 + 发 v0.4.1
```

## 4. 验收标准

| 指标 | 目标 |
|------|------|
| 启动时间（冷） | 比 v0.4.0 快 ≥ 30%（基线先在 v0.4.0 测出来作 baseline） |
| 5000 条历史搜索响应 | < 100 ms |
| 24 小时持续监听 RSS 增长 | < 50 MB |
| panic 触发不丢数据库 | 100%（手动注入 panic 测试） |
| GitHub Actions CI 时间 | < 15 分钟 |

## 6. 风险与回滚

| 风险 | 缓解 |
|------|------|
| C7 `opt-level = "z"` 引入隐藏 bug | benchmarks + 手动冒烟，发现问题立即回 `"s"` |
| C4 启动并行化导致 race condition | 严格用 `tokio::join!` 等待，不 fire-and-forget；加单元测试 |
| C6 panic_hook 写文件本身崩溃造成无限递归 | hook 内部用 `std::panic::catch_unwind` 兜底 |
| A2 卸载弹窗在静默卸载场景挡道 | 检测命令行参数 `/S`（NSIS 静默标志）时跳过弹窗 |
